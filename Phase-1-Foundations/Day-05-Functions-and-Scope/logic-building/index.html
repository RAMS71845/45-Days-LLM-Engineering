Softpro School of AI · Day 5
Logic Building · 30 minutes

How to Build
Logic When You Code

Syntax you can Google. The real skill is turning a problem into a plan a computer can follow. That skill is learnable — here's the method.

Think first. Type second.
The real bottleneck

You know the syntax. You're stuck anyway.

Most beginners can read every line of Python and still stare at a blank screen when handed a new problem. That's normal — and it's not a syntax gap.

It's a logic gap: the leap from "what is being asked" to "what steps solve it." Today is a repeatable method to cross that gap every time.

Definition

Logic building = designing the steps

🧠 The thinking job

Break a problem into a precise, ordered sequence of steps — clear enough that a machine (with zero common sense) can follow it.

⌨️ The typing job

Translate those steps into Python. This is the easy part once the steps exist.

A computer does exactly what you say, in order, literally. Logic building is writing instructions for the world's fastest, most literal-minded helper.

The method

5 steps — and 4 are away from the keyboard

  • 1 · Understand — pin down the inputs, the output, and the rules.
  • 2 · Example by hand — solve one case on paper; watch how your brain does it.
  • 3 · Plan — write the steps in plain English (pseudocode); pick a pattern.
  • 4 · Code — translate each line of the plan into Python.
  • 5 · Test — run normal, boundary, and weird inputs; fix and repeat.

If you jump straight to step 4, you're guessing. The plan is what you're really paid for.

Step 1 of 5

Understand the problem

Don't write a line until you can answer three questions out loud:

📥 Input

What do I get, and in what form? A number? A string? A list?

📤 Output

What exactly must I produce or return? One value? Many?

📏 Rules

What are the conditions and the tricky edges (empty, zero, negative)?

Restate it in one sentence in your own words. If you can't, you don't understand it yet — and no code will fix that.

Step 2 of 5

Do one example by hand

Take a real input and solve it on paper — no code. Then ask the magic question:

The magic question

"What did my brain just do, step by step, to get that answer?"

Your hand-solution is the algorithm — you just have to slow down and notice the steps you did automatically. Do a second example if the first felt like a fluke.

Step 3 of 5

Plan in plain English (pseudocode)

Write the steps, not the syntax
get the number N total = 0 for each number from 1 to N: add it to total show total
Why bother
  • Catches flawed logic before you fight the syntax.
  • Language-independent — same plan works in any language.
  • The comments for your code are already written.
Step 3, continued

The pattern toolbox

Most problems are a few familiar shapes combined. Learn to spot them:

➕ Accumulator

A running total or string you add to in a loop (sum, build text).

🔢 Counter

Tally how many times something happens (count += 1).

🚩 Boolean flag

Track yes/no across a loop ("have I seen one yet?").

🔎 Search + break

Scan until you find it, then stop early.

🏆 Best so far

Keep the current champion (max, min, longest word).

🧩 Decompose

Split into small functions and solve each alone.

Step 4 of 5

Translate the plan to code

Plan
total = 0 for each n from 1 to N: add n to total return total
Python
def add_up(N): total = 0 for n in range(1, N+1): total += n return total

Each plan line becomes one or two lines of code. Now the syntax from Days 1–5 is all you need: variables, loops, conditionals, functions.

Step 5 of 5

Test like you're trying to break it

Code that runs once on one input isn't done. Always try three kinds of input:

  • Normal — a typical case you expect (e.g. N = 5).
  • Boundary — the edges: 0, 1, empty string, the first/last item.
  • Weird — negatives, huge values, the unexpected.

A failing test isn't failure — it's a free bug report. Trace it back to the plan, fix the logic, run again.

When it breaks

Stuck? Debug the logic, not the keyboard

🖨️ Print to see

Drop in print() to watch variables change. Reality vs. what you assumed — the gap is the bug.

🦆 Explain it aloud

Talk through your code line by line (to a friend, or a rubber duck). You'll hear the flaw yourself.

✂️ Shrink the input

Make it the smallest failing case. Less noise, faster fix.

↩️ Back to the plan

Most bugs are wrong logic, not wrong syntax. Re-check step 3, not the semicolons.

Avoid these

Beginner traps

  • Coding before understanding — typing to "figure it out" wastes the most time.
  • Trying to write it perfectly in one go — get it working, then improve it.
  • Copying code you don't understand — it breaks the moment the problem shifts.
  • Ignoring edge cases — empty, zero, and negative inputs are where bugs hide.
  • Do instead: plan on paper, build in small steps, test as you go.
It's a muscle

How logic building actually grows

Nobody is "born logical." It's pattern recognition built by reps — the more problems you solve, the faster you see the shape of a new one.

📅 A little, daily

One or two problems a day beats a weekend binge. Consistency wins.

✍️ Plan first, always

Force the pseudocode step even when you "know" the answer.

🔁 Re-solve

Redo old problems a different way. New angles deepen the patterns.
