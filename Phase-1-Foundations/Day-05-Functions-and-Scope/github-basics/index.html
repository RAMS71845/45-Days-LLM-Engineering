The one idea that confuses everyone

Your code passes through 3 areas

1️⃣ Working directory

Your actual files, as you edit them right now.

2️⃣ Staging area

The "shortlist" — changes you've picked to go in the next snapshot.

3️⃣ Repository

The committed history — snapshots saved forever.

Think of packing a parcel: edit things → put chosen items in the box (stage) → seal & label it (commit) → ship it (push).

The whole workflow, one picture

add → commit → push

add picks what goes in · commit saves the snapshot locally · push uploads it to GitHub.

Two ways to start a project

First, get a repo on your machine

🆕 Brand-new project

Tell GitHub who you are once, then turn a folder into a repo:

# one-time, per computer
$ git config --global user.name "Your Name"
$ git config --global user.email "you@email.com"

# inside your project folder
$ git init

📥 Existing online repo

Copy one down from GitHub to your computer:

$ git clone https://github.com/
   you/your-project.git

# creates a folder, fully wired
# to GitHub — ready to work
The loop, in real commands

Your everyday 4 steps

$ git status # 1. what changed? (run this constantly)
$ git add . # 2. stage all changes (the shortlist)
$ git commit -m "Add login validation" # 3. snapshot + message
$ git push # 4. upload to GitHub

That's the whole job, 90% of the time. add → commit are local; push is the only step that touches the internet.

The command you'll run most

git status = "where am I?"

When in doubt, status. It tells you what's changed, what's staged, and what to do next — it even prints the commands.

$ git status
Changes not staged for commit:
  modified: loops.py
Untracked files:
  notes.txt

# Git is literally telling you: run `git add` next

No such thing as running git status too often. It never changes anything — it just informs.

A small habit that screams "pro"

Write commit messages people can read

❌ Useless

"stuff"
"asdf"
"fixed it"
"final changes ok now"

→

✅ Clear

"Add BMI calculator exercise"
"Fix off-by-one in range loop"
"Update README run steps"

Rule of thumb: finish the sentence "This commit will…" → "…Add login validation." Present tense, what & why. Future-you reads these to time-travel.

Some things must NEVER be pushed

.gitignore — keep secrets & junk out

A file named .gitignore lists what Git should not track. Critical in the AI track — your API keys must never reach GitHub.

# .gitignore
.env # your secret API keys — NEVER commit these
__pycache__/ # Python's cache junk
venv/ # the virtual environment
*.log # log files

⚠️ A leaked API key in a public repo can be abused within minutes. Add .env to .gitignore before your first push.

When you (or a teammate) edit elsewhere

git pull before you start

  • Pull downloads the latest commits from GitHub into your local repo.
  • Edited on another laptop, or on github.com directly? Pull first so you're up to date.
  • Habit: pull → work → add → commit → push. Start in sync, end in sync.
$ git pull # grab the newest changes before you begin
